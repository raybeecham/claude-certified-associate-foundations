# Engineering Patterns

The repository's shared [pattern library](../../patterns/) contains reusable designs derived from the certification lessons and this playbook.

## Pattern format

Each pattern should include:

1. **Name**
2. **Problem**
3. **Context**
4. **Forces and trade-offs**
5. **Recommended design**
6. **Controls**
7. **Failure modes**
8. **Decision rule**
9. **Example**
10. **Related material**

## Planned vendor-neutral patterns

- evidence-first research;
- staged executive briefing;
- structured extraction with exceptions;
- tiered model routing;
- deterministic calculation boundary;
- human approval gate;
- state-capsule handoff;
- retrieval with authority separation;
- controlled tool execution;
- model migration with rollback; and
- evaluation-driven optimization.

## Rule

A pattern should solve a recurring design problem. It should not merely describe a product feature.
