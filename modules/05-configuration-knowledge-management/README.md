# Module 5: Configuration & Knowledge Management

## Why this domain matters

Reusable instructions and knowledge become operational dependencies. Without authority, provenance, freshness, access control, and source conflict rules, a helpful prototype can silently produce stale or unauthorized answers.

## Learning objectives

By the end of this module, you should be able to:

- define instruction precedence and reusable configuration;
- maintain a governed knowledge-source register;
- select the smallest relevant, permitted, authoritative context;
- handle source conflicts and stale information;
- use caching only for stable, reusable prefixes under appropriate data policy;
- keep credentials out of prompts and repositories;
- apply least privilege to retrieval and connectors; and
- test knowledge freshness, provenance, and access boundaries.

## Exam lens

More context is not automatically better. Prefer authoritative, current, scoped evidence. Durable behavior belongs in governed configuration, while sensitive secrets belong in a secret manager and must never be treated as prompt text.

## Completion criteria

- [ ] I can build a source register.
- [ ] I can resolve or surface conflicting sources.
- [ ] I can explain when retrieval and caching help.
- [ ] I can apply least privilege to knowledge access.
- [ ] I completed the knowledge lab and scored at least 80% on the quiz.
