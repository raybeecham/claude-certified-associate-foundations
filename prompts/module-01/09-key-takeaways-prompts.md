# Module 1 Prompt Notebook: Key Takeaways

These prompts help review, teach back, and apply the five durable principles from Module 1. Use fictional, synthetic, public, or authorized information.

## 1. Five-takeaway teach-back

### Use when

You want to test whether you can explain Module 1 without relying on memorized wording.

```text
Act as a certification coach.

Ask me to explain these five principles one at a time:
1. select the entry point before writing the prompt;
2. separate Project context, Skill procedure, Code Execution, and Memory;
3. match Haiku, Sonnet, and Opus to the task;
4. treat context as a managed budget; and
5. treat review as a structural part of generative-AI work.

For each response:
- score accuracy from 0 to 2;
- identify missing distinctions;
- ask one scenario-based follow-up;
- correct misconceptions; and
- provide a concise model answer only after I respond.

Do not award full credit for naming a feature without explaining why it fits.
```

## 2. Integrated platform selector

### Use when

You need to apply all four Module 1 decisions to one task.

```text
Act as an AI workflow architect.

Task:
[DESCRIBE TASK]

Recommend:
1. entry point: Chat, Project, Artifact, Research, or a combination;
2. capability layer: Project instructions, Project knowledge, Skill, Code Execution, Memory, or none;
3. model tier: Haiku, Sonnet, or Opus under the certification framework;
4. context strategy: current thread, Project configuration, Memory, state capsule, or authoritative external record; and
5. controls: source verification, deterministic validation, escalation, and human review.

For each recommendation, identify:
- scenario signal;
- benefit;
- cost or operational burden;
- plausible alternative;
- why the alternative is weaker; and
- condition that would change the answer.

Prefer the least complex configuration that meets the requirement.
```

## 3. Entry-point challenge

### Use when

You want focused practice distinguishing Chat, Projects, Artifacts, and Research.

```text
Generate eight original professional scenarios.

For each scenario, ask me to choose among:
- Chat;
- Project;
- Artifact;
- Research; or
- a combination.

Ensure the scenarios include:
- a one-off task;
- recurring work with stable context;
- an editable deliverable;
- a recent multi-source investigation;
- a recurring investigation producing a deliverable;
- a task that looks recurring but is actually temporary;
- a task that looks like Research but needs only uploaded sources; and
- an over-engineered Project candidate.

After I answer, explain the controlling signal and why each alternative is weaker.
```

## 4. Capability placement drill

### Use when

You need to separate context, procedure, computation, continuity, and authority.

```text
Generate twelve workflow components and ask me to classify each as:
- Project standing instructions;
- Project knowledge;
- current task input;
- Skill;
- Code Execution;
- Memory;
- authoritative external record;
- human review; or
- exclude.

Include close calls involving:
- reusable versus cycle-specific files;
- behavioral rules versus procedures;
- preferences versus workstream instructions;
- calculations versus analytical judgment;
- remembered facts versus current authoritative state; and
- automated output versus accountable approval.

After I classify each item, provide a rationale and lifecycle explanation.
```

## 5. Model-tier trade-off coach

### Use when

You need to practice defending Haiku, Sonnet, or Opus.

```text
Give me six original scenarios that vary by:
- structure;
- ambiguity;
- volume;
- consequence;
- latency requirement;
- cost or usage constraint; and
- availability of validation and escalation.

For each scenario, require one sentence that states:
1. the selected tier;
2. what the tier buys; and
3. what the tier costs.

Score each answer from 0 to 2 on:
- correct tier;
- use of scenario signals;
- explicit benefit;
- explicit cost; and
- explanation of when another tier would be appropriate.

Remind me that production selection requires representative evaluation.
```

## 6. Context intervention simulator

### Use when

You need to distinguish restart, summarize, and persist.

```text
Present eight context-management situations.

For each, ask me whether to:
- continue normally;
- clean the current thread;
- summarize;
- persist;
- restart;
- summarize, persist, and restart;
- investigate another root cause; or
- stop because the data or scope is unauthorized.

Include cases involving:
- lost early instructions;
- a genuinely new project phase;
- a stale Project document;
- an underpowered model;
- a long but still coherent conversation;
- a usage limit rather than a context limit;
- a summary that contains an unverified claim; and
- unrelated confidential workstreams.

Explain the diagnosis before the remedy.
```

## 7. Review-control mapper

### Use when

You want to identify what remains outside the model.

```text
Review the workflow below.

Workflow:
[DESCRIBE WORKFLOW]

Identify which controls must remain outside Claude:
- source authority;
- identity and authorization;
- data classification;
- deterministic calculation;
- schema validation;
- current operational state;
- approval authority;
- logging and audit;
- retention and deletion; and
- rollback.

For each control, state:
1. why model capability does not replace it;
2. where the control should live;
3. who owns it; and
4. how it should be tested.
```

## 8. Distractor-analysis drill

### Use when

You want to improve scenario-question judgment.

```text
Generate five original Module 1 multiple-choice questions.

For each question:
- provide four plausible options;
- make at least two options technically possible but suboptimal;
- identify one controlling scenario signal;
- require me to select the best answer; and
- require me to explain why every distractor is weaker.

After I answer, classify any mistake as:
- entry-point error;
- capability-placement error;
- model-tier error;
- context-management error;
- current-information error;
- computation error;
- authority or human-review error; or
- over-engineering.
```

## 9. One-page cheat-sheet builder

### Use when

You want a personalized review page from your weak areas.

```text
Create a one-page Module 1 cheat sheet based on the weak areas below.

Weak areas:
[LIST WEAK AREAS]

Include:
- one decision table;
- one integrated workflow sequence;
- eight high-value distinctions;
- five anti-patterns;
- five flashcards;
- three scenario signals I should watch for; and
- one final readiness checklist.

Keep product-specific model names, context sizes, pricing, and usage allowances in a dated note rather than treating them as permanent facts.
```

## 10. Module 1 oral examination

### Use when

You want a rigorous final review before closing the module.

```text
Conduct a ten-question oral examination on Claude Platform and Model Foundations.

Cover:
- generative variation;
- entry-point selection;
- Project instructions versus knowledge;
- Skills;
- Code Execution;
- Memory;
- Haiku, Sonnet, and Opus;
- context degradation;
- usage limits versus context limits; and
- human review.

Ask one question at a time.

For every answer:
- require a reason;
- require a trade-off;
- ask what control remains outside the model;
- score from 0 to 2; and
- keep a running total.

Passing standard:
- at least 16 out of 20;
- no zero on Code Execution, context recovery, or human review; and
- ability to explain why a plausible alternative is weaker.
```

## Suggested study sequence

1. Run the five-takeaway teach-back.
2. Complete the capability placement drill.
3. Complete the model-tier trade-off coach.
4. Complete the context intervention simulator.
5. Finish with the oral examination.
